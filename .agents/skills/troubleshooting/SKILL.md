# Troubleshooting Skill

이 스킬은 시스템의 오동작이나 외부 환경 변화(API 변경 등)로 인한 문제를 체계적으로 진단하고 해결하기 위한 프로토콜을 제공합니다.

## 진단 프로토콜 (Diagnosis Protocol)

문제가 보고되면 에이전트는 다음 단계를 순차적으로 실행합니다:

### 1단계: 현상 파악 및 데이터 수집
- **로그 분석**: 최근 에러 로그나 시스템 로그를 확인하여 발생 시점과 패턴 파악.
- **상태 확인**: 현재 시스템의 설정값, DB 상태, 외부 API 응답 값 수집.
- **재현 조건**: 문제가 발생하는 특정 조건(특정 키워드, 특정 시간대 등) 식별.

### 2단계: 변경점 분석 (Change Analysis)
- **외부 환경**: API 명세 변경, 매체사 정책 변경 사항 검색 및 대조.
- **내부 코드**: 최근 수정된 코드(Git Diff 등)와 문제 발생 시점의 상관관계 분석.
- **버전 비교**: 작동하던 이전 버전의 로직과 현재 로직의 차이점 식별.

### 3단계: 가설 수립 및 검증
- **가설 1**: "API 응답 필드명이 변경되어 데이터 파싱에 실패했을 것이다."
- **가설 2**: "순위 계산 알고리즘의 임계값이 현재 시장 상황과 맞지 않는다."
- **검증**: 가설을 확인하기 위한 최소 단위의 테스트 코드 실행 또는 데이터 시뮬레이션.

### 4단계: 해결책 도출 및 검토
- **수정안 제안**: 코드 수정, 설정 변경, 또는 인프라 조정안 작성.
- **영향도 평가**: 수정 시 발생할 수 있는 사이드 이펙트(Side Effect) 체크.
- **장기 대책**: 동일 문제 재발 방지를 위한 모니터링 강화 방안 제안.

## 활용 예시

- **API 변경 대응**: "네이버 쇼핑 API 결과값이 이전과 달라져서 순위 보정이 안 됨" -> 1단계(응답값 수집) -> 2단계(API 공지 확인) -> 3단계(파싱 로직 수정 가설) -> 4단계(코드 수정안).
- **데이터 불일치**: "스크래핑 순위와 서버 순위가 다름" -> 1단계(양측 데이터 샘플 추출) -> 2단계(추출 방식 차이 분석) -> 4단계(보정 계수 조정).

## 산출물

- `troubleshooting_report.md`: 문제 원인, 분석 과정, 가설 검증 결과, 최종 수정 제안 포함.
